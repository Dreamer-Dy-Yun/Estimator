# 후보군 담기 및 스냅샷 흐름 변경 정리

- 작성 방법 지시: 사용자
- 작성자: Codex
- 작성일: 2026-05-13

## Goal

자사/경쟁사 분석은 "상품을 후보군에 담는 입구"로 단순화하고, 이너후보군은 "조회 기간 기준 수치 검토 및 스냅샷 확정"의 책임을 가진다.

## Scope

- 자사 분석 리스트
- 경쟁사 분석 리스트
- 공통 분석 필터
- 공통 ProductDrawer
- 이너후보군 상세 모달
- 후보군 API 계약 및 mock API
- 스냅샷 저장/표시 문서

## Principles

- 자사/경쟁사 탭에서는 2차 드로워를 열지 않는다.
- 2차 드로워 코드는 삭제하지 않고, 호출자가 열 수 있는지 여부를 매개변수로 제어한다.
- 후보군에 담기는 스냅샷 저장이 아니라 SKU 후보 등록이다.
- 스냅샷은 이너후보군 2차 드로워에서만 저장/수정한다.
- 이너후보군 리스트의 수치는 데이터 참조 기간에 따라 다시 계산되는 값이다.
- 상세확정 여부는 스냅샷 존재 여부로 판단한다.
- SKU 식별은 `code + colorCode + size` 기준을 따른다.

## Confirmed Interpretation

- 색상 필터는 브랜드, 카테고리, 품번, 상품명 필터와 같은 공통 필터 컴포넌트를 사용한다.
- 색상 필터 옵션 순서는 mock 색상 순서와 동일한 `010, 020, 030, 100, 110, 200, 210, 300` 순서를 따른다.
- 자사/경쟁사 분석에서 2차 드로워 여는 반원 버튼은 렌더링하지 않는다.
- 이너후보군에서만 2차 드로워와 스냅샷 저장 기능을 계속 사용한다.

## Confirmed Details

- 선택한 물품을 후보군으로 담을 때는 스냅샷 없이 `stashUuid + productId`만 저장한다.
- 현재 프론트의 `productId`는 내부적으로 `SKU.code + SKU.color_code`에 대응하는 상품 단위로 취급한다.
- 좌/우 방향키는 `좌=열기`, `우=닫기`로 확정한다. 드로워가 오른쪽에서 왼쪽으로 나오는 동선 기준이다.
- 드로워가 열린 상태의 상/하 방향키는 이전/다음 리스트 아이템 이동이다.
- 자사/경쟁사에서는 방향키로도 2차 드로워를 열지 않는다. 2차 드로워는 이너후보군에서만 연다.
- 라이브 계산 화면은 AI 코멘트와 사이즈별 오더량 중심으로 보되, 스냅샷 보기 상태에서는 그래프 외 저장 당시 전체 값과 기간을 표시한다. 그래프는 해당 기간 기준으로 다시 계산해 표시한다.

## API Notes

- 이너후보군 리스트는 조회 기간 변경마다 백엔드 계산이 발생할 수 있다.
- 백엔드는 해당 조회 기간의 전체 상품 데이터를 먼저 집계하고, 그 전체 분포 기준으로 배지를 부여한 뒤, 그 중 해당 후보군에 담긴 상품만 추려 반환해야 한다.
- 백엔드는 후보군 아이템 목록 조회에서 기간 기반 집계/추천 수치를 함께 계산하거나, 후보군 아이템 정적 정보와 기간 집계 수치를 분리 제공할 수 있다.
- 프론트 API 호출부에는 해당 호출이 기간 변경마다 재실행될 수 있음을 명시한다.
- 백엔드 부하가 커질 경우 추천 방식은 다음 중 하나를 고려한다.
  - 후보군 아이템 정적 목록과 기간 집계 API 분리
  - 기간별 계산 결과 캐시
  - 기간+경쟁채널 기준 전체 랭킹/배지 결과를 캐시하고 후보군 상세 조회는 그 결과와 stash item을 조인
  - 조회 기간 입력 debounce 또는 명시적 조회 버튼
  - 서버 측 batch 계산

## Result Target

- 사용자는 분석 리스트에서 여러 상품을 체크하고 후보군에 담는다.
- 사용자는 이너후보군에서 데이터 참조 기간을 바꿔 리스트 수치를 재계산해 본다.
- 사용자는 이너후보군 2차 드로워에서 사이즈별 발주량을 조정하고 스냅샷을 저장한다.
- 리스트는 스냅샷 존재 여부를 상세확정 컬럼으로 표시한다.
