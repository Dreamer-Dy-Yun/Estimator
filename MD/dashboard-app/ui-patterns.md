## 목적

이 문서는 화면 디자인을 새로 정하는 문서가 아니라, 기존 `dashboard-app`의 카드/그리드/패널 리듬을 유지하기 위한 **공통 UI 계약**이다. 새 화면이나 공통 컴포넌트를 만들 때 이 문서를 먼저 확인한다.

## 버튼

- 같은 의미 계층의 버튼은 높이와 기본 폭을 맞춘다.
- 버튼 텍스트는 가능한 한 한 줄로 유지한다.
- 위험 action은 기존 danger 톤과 아이콘 패턴을 따른다.
- 로딩 중 버튼은 기존 크기를 유지하고 내부 spinner 또는 텍스트 변경으로 상태를 표시한다.
- 주요 탭과 유틸리티 action은 시각 계층을 분리한다.
- 단순 텍스트 버튼보다 의미가 명확한 icon+label 버튼을 우선한다.

## 카드와 패널

- 화면 섹션을 카드 안의 카드로 중첩하지 않는다.
- 카드 radius, border, background는 기존 CSS Modules 톤을 따른다.
- 정상 완료 상태를 위해 별도 카드를 계속 차지하게 두지 않는다.
- 리스트가 많아지는 패널은 헤더와 본문을 분리하고, 본문만 스크롤한다.
- 패널 안 action 영역은 기능 단위로 묶고, 서로 다른 기능을 같은 버튼 그룹처럼 보이게 하지 않는다.

## 리스트와 테이블

- 헤더는 컬럼 의미를 정확히 말해야 한다.
- 정렬 가능한 헤더는 `button` 또는 접근 가능한 interactive element를 사용한다.
- 표시 인덱스는 현재 정렬/필터 결과의 화면 순서 기준이다.
- 체크박스 컬럼은 체크박스 row와 헤더 위치를 맞춘다.
- 컬럼 헤더는 데이터 성격에 맞춘다. 문자열 식별 컬럼은 왼쪽, 상태/배지성 컬럼은 중앙, 숫자/금액 컬럼은 오른쪽 정렬한다.
- 모바일 수준이 아닌 뷰포트에서는 불필요한 내부 가로 스크롤을 피한다.
- 단, 이너 후보 리스트처럼 운영 컬럼 수가 많고 값 비교가 우선인 고정 데이터 테이블은 예외다. 이 경우 헤더 고정, 본문 스크롤, 동적 폰트 축소로 탐색성을 유지한다.
- 데이터가 로딩 중인 셀은 다른 값으로 대체하지 않고 로딩/실패 상태를 표시한다.

## 드로워와 모달

- 드로워/모달 헤더는 현재 레이어의 목적을 말해야 한다.
- ESC, 좌/우 방향키, 닫기 버튼의 의미는 화면별 QA 계약과 일치해야 한다.
- 드로워 바깥의 실제 버튼/입력/토글 클릭은 바깥 클릭 닫힘보다 우선한다.
- 메타 정보는 한 줄 말줄임을 기본으로 하고, 전체 값은 tooltip 또는 `title`로 확인할 수 있어야 한다.
- action 버튼 영역은 문구 길이에 따라 개행되지 않게 폭과 padding을 조정한다.
- snapshot/live 전환처럼 데이터 기준이 바뀌는 토글은 다른 destructive action과 같은 그룹에 두지 않는다.

## Toast와 상태 표시

- mutation 성공/실패는 상단 자동 닫힘 toast로 알린다.
- 긴 요청 진행률은 toast보다 진행 팝업이나 row 단위 상태가 낫다.
- 기본 데이터 정상 조회 완료 상태는 계속 노출하지 않는다.
- 오류와 권한 실패는 사용자가 원인을 추측하지 않도록 구체적으로 표시한다.

## Tooltip과 말줄임

- 식별자, 상품명, 후보군명, API 키 이름처럼 중요한 값이 잘리면 tooltip으로 전체 값을 제공한다.
- 긴 텍스트 때문에 row 높이가 예측 불가능하게 늘어나면 말줄임을 우선한다.
- tooltip은 숨겨진 비즈니스 값을 새로 만들지 않고, 이미 받은 값을 표시한다.

## 반응형 기준

- 모바일 수준에서는 세로 stack을 허용한다.
- 데스크톱/태블릿 수준에서는 폰트, 컬럼 폭, action 폭을 조정해 불필요한 가로 스크롤을 줄인다.
- 체크박스, 닫기, 삭제 같은 핵심 터치 타깃은 지나치게 작게 줄이지 않는다.
- 드로워가 열린 상태에서도 원본 리스트의 현재 row와 포커스가 사용자가 추적 가능한 방식으로 유지되어야 한다.

## 공용화 기준

- 사용자 관리, GPT 키 관리, 구글 시트 관리처럼 구조가 같은 화면은 공통 shell을 쓴다.
- 상태 표시, 리스트 스크롤, 패널 헤더, action bar가 세 화면 이상 반복되면 공통 컴포넌트 후보로 본다.
- 단, 의미가 다른 action을 추상화 때문에 같은 컴포넌트 props로 억지로 합치지 않는다.
