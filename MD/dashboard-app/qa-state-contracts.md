## 목적

이 문서는 화면별 세부 QA가 아니라, 모든 화면에서 공통으로 지켜야 하는 **정상/비정상 상태 표시 계약**을 정의한다. 상태 모델이나 API 실패 처리가 바뀌면 코드, API 문서, 이 문서를 같은 변경 단위에서 갱신한다.

## 공통 원칙

- 정상 idle 상태는 화면 공간을 차지하지 않는다.
- 사용자가 기다려야 하거나 판단해야 하는 상태만 표시한다.
- 이전 데이터가 남아 있는 갱신 중 상태는 빈 화면보다 낫지만, 반드시 갱신 중임을 표시한다.
- 백엔드가 제공해야 하는 비즈니스 값은 프론트가 다른 값으로 조용히 대체하지 않는다.
- 권한 실패, API 실패, stale 응답, 사용자 입력 오류는 숨기지 않는다.

## 상태별 표시 계약

| 상태 | 기준 표시 | 금지 |
|------|-----------|------|
| 최초 로딩 | 화면 또는 리스트 영역에 spinner를 표시한다. | 빈 결과처럼 보이게 두지 않는다. |
| 재조회/갱신 중 | 기존 데이터를 유지하고 compact spinner 또는 row/column 단위 로딩을 표시한다. | 기존 리스트를 비워 깜빡이게 하지 않는다. |
| 빈 결과 | 명시적인 빈 상태 문구를 표시한다. | 오류처럼 보이게 하지 않는다. |
| API 실패 | toast, panel message, row failure 중 하나로 실패를 표시한다. | 성공처럼 조용히 무시하지 않는다. |
| 권한 실패 | `401`은 로그인 흐름, `403`은 권한 부족 상태로 드러낸다. | 관리자 기능을 보이게 둔 뒤 조용히 실패시키지 않는다. |
| stale 응답 | 최신 요청 sequence보다 오래된 응답은 화면을 덮지 않는다. | 늦게 온 이전 요청으로 최신 화면을 되돌리지 않는다. |
| mutation 성공 | 자동 닫힘 toast로 완료를 알리고, 필요한 경우 로컬 상태를 즉시 갱신한다. | 성공 후 불필요한 전체 재조회로 stale 상태를 되살리지 않는다. |
| mutation 실패 | 실패 toast와 현재 상태 유지를 기본으로 한다. | 실패했는데 성공한 것처럼 로컬 상태만 바꾸지 않는다. |
| 비활성 action | disabled 상태와 이유가 UI 문맥에서 이해되어야 한다. | 클릭 가능한 것처럼 보이게 두지 않는다. |
| 긴 텍스트 | 한 줄 말줄임과 tooltip으로 전체 값을 확인할 수 있게 한다. | 핵심 식별자를 임의 개행해 레이아웃을 깨지 않는다. |
| 부분 로딩 | 셀, row, column 단위로 `로딩중` 또는 spinner를 표시한다. | 관련 없는 값으로 임시 대체하지 않는다. |

## SSE/비동기 작업

| 구분 | 기준 |
|------|------|
| 시작 | job 생성 성공 후 진행 상태 팝업 또는 row 단위 로딩을 표시한다. |
| 진행 | item 완료/실패 이벤트는 해당 row만 갱신한다. |
| 완료 | 요청한 item이 모두 완료/실패되면 연결을 닫는다. |
| 실패 | 전체 실패와 item 실패를 구분해 표시한다. |
| stale | 기간/필터/대상 변경 전 job 이벤트는 현재 화면을 덮지 않는다. |
| 재시도 | 같은 URL 자동 재접속으로 중복 요청이 생기면 안 된다. |

## API 실패 분류 기준

| 종류 | 예시 | UX 기준 |
|------|------|---------|
| 인증 실패 | `401` | 로그인 페이지 이동 또는 세션 만료 안내 |
| 권한 실패 | `403` | 권한 부족 메시지와 action 비활성 |
| 찾을 수 없음 | `404` | 대상 없음 또는 삭제됨 안내 |
| 검증 실패 | `422` | 사용자 입력/요청 조건 오류 표시 |
| 충돌 | `409` | 최신 상태 재확인 또는 충돌 안내 |
| 클라이언트 실패 | 그 외 `4xx` | 요청 조건이나 화면 흐름 확인 안내 |
| 서버 실패 | `500` 이상 | 재시도 가능한 실패 안내 |
| 네트워크 실패 | fetch reject, SSE transport error | 연결 실패 안내와 기존 데이터 유지 |

## 변경 체크리스트

- [ ] 이 상태가 정상 idle인지, 사용자가 알아야 하는 상태인지 구분했다.
- [ ] 기존 데이터를 유지할지 비울지 기준을 정했다.
- [ ] 실패를 숨기는 fallback이 없는지 확인했다.
- [ ] mutation 성공 후 로컬 반영과 서버 재조회 순서를 검토했다.
- [ ] 권한 실패와 일반 실패를 같은 문구로 뭉개지 않았다.
- [ ] `qa-current-behavior.md`의 화면별 QA와 충돌하지 않는다.
