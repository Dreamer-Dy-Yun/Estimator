# common-rules.md

## 공통 원칙

- 역할 분리는 `책임 단위 기준`으로 하며, `파일 단위`가 아니라 `작업 목적 단위`가 우선이다.
- 각 문서는 실제 수정 권한을 명시하고, Sub-Agent는 해당 범위 밖의 파일을 건드리지 않는다.
- 같은 파일을 병렬로 수정하지 않는다. 동일 파일 후속 작업은 `done` 처리 후 순차 진행한다.
- 경계 밖 수정은 즉시 중단하고 QA에 이슈로 반영한다.
- 작업 결과는 `review/`을 거쳐 `done/`으로만 승인한다.

## 파일 접근 권한

각 todo는 다음 항목을 반드시 포함한다.

- 수정 가능 파일: 해당 Sub-Agent가 직접 수정 가능한 파일
- 생성 가능 파일: 해당 Sub-Agent가 새로 만들 수 있는 파일
- 읽기 전용 파일: 읽기만 가능하고 수정·삭제 금지인 파일
- 수정 금지 파일: 읽기, 수정, 삭제 모두 금지인 파일

모든 역할은 `수정 가능 파일`만 실제 변경 대상이다.

## 동일 파일 수정 제한

- 하나의 todo 안에서라도 충돌이 발생할 수 있는 파일은 동시 배정하지 않는다.
- 동일 파일을 여러 Sub-Agent가 수정해야 하는 경우:
  - 방식 1: 단일 Sub-Agent가 통합 처리
  - 방식 2: 선행 작업으로 책임 분리 후 순차 분해
  - 방식 3: 선행 후속 todo로 순차 작업

## 작업 단위 기준

- todo 1개는 1개 목적, 1개 책임 영역으로 제한한다.
- 1개 todo는 `선행 조건`, `수정 가능 파일`, `완료 기준`을 필수로 포함한다.
- 범위가 커지면 기능 단위로 더 작게 쪼갠다.

## 입력/출력 계약 기준

- Sub-Agent 간 연결 필요 시 입력/출력 항목을 todo에 정의한다.
- 입력은 읽기 대상, 의존 함수/클래스, 보호할 인터페이스를 명시한다.
- 출력은 생성/수정 파일, 반환 형태, 유지해야 할 호환성 범위를 명시한다.
- 계약은 plan/todo 단계에서 먼저 고정하고 임의 변경은 금지한다.

## review 필수 항목

review 문서는 아래 항목을 모두 포함해야 한다.

- 수행 일시
- 참조한 todo
- 수행 내용
- 변경 파일
- 생성 파일
- 검증 내용
- 남은 이슈
- QA 확인 요청 사항

## done 처리 원칙

- QA가 완료 기준 충족 여부를 검증해야 하며, 충족 시 `done/` 이동으로 종료한다.
- 불충분하면 todo 재작성 또는 plan 재정의로 회귀한다.
- 완료 판단 시 선행 조건, 범위 일치, 권한 준수, 하위 계약 변경 유무를 함께 확인한다.

## 금지 사항

- todo에 없는 파일 수정
- 권한 범위를 벗어난 수정·생성
- 동일 파일 다중 수정(동시)
- 선행 조건이 미완료인 후속 작업 수행
- review 없이 done 처리
- QA 승인 전 todo/plan 임의 변경
