# orchestrator.md

## plan 참조 규칙

- plan 문서의 요구사항/설계 의도를 먼저 확인한다.
- plan의 책임 변경, 계약 변경, 폴더 경계 변경 여부를 추출해 todo로 분해한다.
- plan과 입력 계약이 불충분하면 todo 작성 전에 plan에 보강안을 작성한다.

## todo 작성 규칙

- todo 파일명은 `TODO-###.md` 형식으로 생성한다.
- 각 todo는 목적, 작업 범위, 선행 조건, 파일 권한, 완료 기준을 반드시 포함한다.
- 변경 의존성이 큰 항목은 작은 단위로 분해한다.

## 작업 경계 정의 규칙

- 1개의 todo는 1개 책임 영역만 다룬다.
- `수정 가능 파일`, `생성 가능 파일`은 교집합이 없도록 설정한다.
- 동일 파일이 여러 todo에 걸릴 때는:
  - 병렬 금지
  - 선행 todo 완료 후 재분해해서 순차 진행

## 파일 접근 권한 정의 규칙

- 각 todo에 대해 아래 4개 범주를 명확히 선언한다.
  - 수정 가능 파일
  - 생성 가능 파일
  - 읽기 전용 파일
  - 수정 금지 파일
- Sub-Agent가 수정 가능한 범위를 넘어서는 변경은 요청 불허로 간주한다.

## 선행 조건 정의 규칙

- 선행 todo가 없으면 `없음`으로 명시한다.
- 선행 조건이 있는 todo는 이름/번호를 정확히 기재한다.
- 선행 조건이 완료되지 않으면 후속 todo는 배정하지 않는다.

## 입력/출력 계약 작성 규칙

- 입력: 참조 파일, 계약 데이터, 기존 인터페이스, 금지 인터페이스를 기재한다.
- 출력: 변경 파일, 생성 파일, 반환 형식, 하위 호환성(필요 시)과 공개 경계 변경점을 기재한다.
- 계약 변경이 동반되면 plan에 먼저 반영 후 todo로 이어간다.

## Sub-Agent 충돌 방지 규칙

- 동일 파일 병렬 수정 감지 금지.
- todo당 단일 책임·단일 파일 집합 원칙을 우선 적용한다.
- 충돌 이슈가 확인되면 해당 todo는 `review`로 이관 전에 중지하고 경계 재정의한다.
